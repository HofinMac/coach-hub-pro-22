## Project overview

**Coach Hub** — a platform for personal trainers/coaches (Czech-language product) to manage clients, training plans, bookings, messaging, payments, and a partner/rewards program. Built with Lovable (React + Vite + shadcn-ui), backed by Supabase (Postgres + Auth).

This repo is synced with Lovable (https://lovable.dev) — changes pushed here are reflected there and vice versa. Files marked `// This file is automatically generated. Do not edit it directly.` or `// This file is auto-generated by Lovable. Do not modify it.` (e.g. `src/integrations/supabase/client.ts`, `src/integrations/supabase/types.ts`, `src/integrations/lovable/index.ts`) should not be hand-edited — regenerate/sync via Lovable or the Supabase CLI instead.

## Architecture

### Stack
- Vite + React 18 + TypeScript, `@vitejs/plugin-react-swc`
- Routing: `react-router-dom` (all routes declared flat in `src/App.tsx`, no nested route config)
- UI: shadcn-ui components in `src/components/ui/**` (generated via `components.json`, aliases `@/components`, `@/lib`, `@/hooks`) + Tailwind CSS with HSL CSS-variable design tokens (see `tailwind.config.ts` / `src/index.css`) — use the semantic tokens (`primary`, `muted`, `success`, `warning`, `destructive`, `sidebar-*`, etc.) rather than hardcoded colors
- Data/server state: `@tanstack/react-query`
- Backend: Supabase (Postgres, Auth, RPC, migrations in `supabase/migrations/`, project config in `supabase/config.toml`)
- PWA: `vite-plugin-pwa` (manifest + Supabase API runtime caching configured in `vite.config.ts`)
- Path alias: `@/*` → `src/*`

### Two parallel app shells (coach vs. client)
The app serves two audiences from one codebase, each with its own layout, nav, and route prefix:
- **Coach** ("trenér") routes are unprefixed (`/dashboard`, `/clients`, `/training`, `/calendar`, `/gyms`, `/messages`, `/payments`, `/benefits`, `/admin/*`, `/settings`) and wrapped in `AppLayout`.
- **Client** ("klient") routes live under `/klient/*` (Czech path segments, e.g. `/klient/treninky`, `/klient/kalendar`, `/klient/platby`) and are wrapped in `ClientLayout`.
- `RoleSwitcher` is a floating UI control that just navigates between `/dashboard` and `/klient` — it does not change auth/role state, so don't treat it as an auth mechanism.
- `TimerPage` is shared between both shells (mounted at both `/timer` and `/klient/casovac`).

### Auth & authorization
- Session/auth via Supabase Auth (`supabase.auth.getUser()`), token storage in `localStorage`.
- Role checks (admin vs. coach) go through the Postgres RPC `get_user_role(_user_id)` — see `AdminGuard.tsx` and `AppLayout.tsx` for the pattern: fetch user, call the RPC, gate UI/redirect based on the result. Admin-only routes (`/admin`, `/admin/partners`, `/admin/campaigns*`, `/admin/approvals`) are wrapped in `<AdminGuard>` inside `<CoachShell>`.
- Custom OAuth/session bridging goes through `src/integrations/lovable/index.ts` (`lovable.auth.signInWithOAuth`), which calls Lovable's cloud-auth SDK and then hands the resulting tokens to `supabase.auth.setSession`.

### Domain model
Core Supabase tables (see `supabase/migrations/`): `profiles`, `user_settings`, `gyms`, `coach_slots`, `slot_bookings`, `slot_share_log`, `coach_benefits`, `coach_certificates`, `client_challenges`, `promo_campaigns`, `campaign_rules`, `promo_codes`, `partners`, `partner_audit_log`, `redemptions`, `reward_history`, `eligibility`.

`src/lib/demo-data.ts` defines the client-side TypeScript domain types (`Coach`, `Client`, `Exercise`, `WorkoutPlan`, `PlanExercise`, `Booking`, `ProgressEntry`, and status/enum unions like `ClientStatus`, `BookingStatus`, `PlanStatus`, `ExerciseCategory`) used across coach-facing features (clients, training plans, bookings, progress tracking).

### Partner/rewards rule engine
`src/lib/partner-engine.ts` implements a small rule engine for campaign eligibility and challenge completion: `RuleType` (`min_attendance`, `min_sessions_month`, `plan_completion`, `course_completion`, `manual_approval`, `streak_days`, `weight_goal`, `custom_metric`) combined with an `Operator` (`eq`/`gt`/`gte`/`lt`/`lte`) against a numeric threshold, producing `RuleResult`/`EligibilityResult`. Extend by adding new `RuleType` values and evaluators here, keeping it in sync with `campaign_rules`/`eligibility` tables.

### Navigation customization
`src/hooks/use-tab-order.ts` backs `TabOrderDialog.tsx` and lets users (per role, "coach" vs. client) reorder/pin bottom-nav tabs, splitting `navItems` into `mainItems` (visible) and `overflowItems` (behind a "More" menu) — both `AppLayout` and `ClientLayout` use this pattern for their mobile nav.

## Testing
- Unit/component tests: Vitest + Testing Library + jsdom, files matched by `src/**/*.{test,spec}.{ts,tsx}`, setup in `src/test/setup.ts`.
- E2E: Playwright, configured via the shared `lovable-agent-playwright-config` package (`playwright.config.ts`); a fixture helper lives in `playwright-fixture.ts`.

## Environment
Requires `VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY`, `VITE_SUPABASE_PROJECT_ID` in `.env` (see `supabase/config.toml` for the linked project id).
